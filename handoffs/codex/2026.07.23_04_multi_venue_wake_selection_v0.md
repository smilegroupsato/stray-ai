## Human decisions — do not reinterpret

1. Selection is completely side-effect-free except for its own append-only selection record.
2. `registry/venues.yml` is the trusted Source of Truth for Venue IDs admitted to selection.
3. v0 includes deterministic body gating and an optional bounded command selector.
4. Selection writes its record and stops before the existing single-Venue wake command.

Do not add impulses or modify state after a selected result.

run_wake_selection(

Exact internal names may differ, but behavior and testability must match the contract.

The CLI must be manual and explicit. Support:

- repeated `--candidate venue_id=snapshot_id`
- `--use-current-snapshots`

Reject using explicit candidates and current-snapshot mode together unless a clearly fail-closed merged-set behavior is specified and tested. Prefer making them mutually exclusive in v0.

### Trusted Venue registry

Load `registry/venues.yml` with strict validation.

Required shape:

```yaml
schema_version: 0.1
venues:
  - venue_id: eternal-free-party
    display_name: Eternal Free Party
    selection_enabled: true
```

Validation must reject the entire run safely when:

- root is not an object
- schema is unsupported
- `venues` is not a list
- entry is not an object
- Venue ID is missing, empty, duplicated, path-like, or not a bounded safe slug
- `selection_enabled` is not boolean
- unknown fields would create ambiguous executable meaning
- enabled Venue count exceeds the bounded maximum

`display_name` is human registry metadata only. Do not send it to the selector payload.

Do not place local paths, remote instructions, source URLs, snapshot metadata, or commands in the registry.

### Candidate construction

Explicit input format:

```text
venue_id=snapshot_id
```

Current-snapshot mode:

- inspect only `<venues-root>/<venue-id>/current`
- require `current` to be a symlink
- require its raw target to be one safe single path component
- resolve only to `<venues-root>/<venue-id>/<snapshot-id>`
- require the final snapshot directory to exist, be a directory, and not itself be a symlink
- never read any file inside it

For explicit candidates:

- require Venue ID to exist and be enabled in the registry
- require a safe single-component opaque snapshot ID
- require the corresponding existing local snapshot directory
- reject duplicates, unknown IDs, malformed values, path separators, absolute paths, dot segments, symlink directories, and path escape

Candidate count must be bounded. A default maximum of 8 is appropriate; hard-bound any profile override if one is introduced.

If any explicit candidate is invalid, the entire candidate set fails closed. Do not silently drop invalid explicit candidates.

For current-snapshot mode, a registered Venue without a usable current symlink may be recorded as a safe omission. It must not be represented as changed or unchanged.

Sort validated candidates by `venue_id` before:

- same-Venue comparison
- selector payload construction
- persisted record construction

Input order, registry order, filesystem order, timestamps, and snapshot recency must not rank candidates.

### Same-Venue comparison

Reuse the current safe semantics in `wake.py`:

- previous snapshot comes only from the latest preserved Visit to the same Venue
- no cross-Venue fallback
- no Venue content read
- `comparison_available` is distinct from `changed`

Do not change current single-Venue wake behavior or historical records.

Package-internal reuse of safe helpers from `wake.py` is acceptable. A small shared-helper extraction is allowed only when existing wake behavior and tests remain unchanged.

Each candidate selector fact must contain only:

```json
{
  "venue_id": "trusted-id",
  "previous_snapshot_id": "opaque-id-or-null",
  "candidate_snapshot_id": "opaque-id",
  "comparison_available": true,
  "comparison_scope": "same_venue_history | same_venue_no_history | explicit_previous",
  "changed": false,
  "content_was_not_read": true
}
```

Explicit-previous support is optional in the first CLI, but the internal schema must not invent a different meaning for the scope.

### Candidate-independent body gate

Before invoking the optional selector, evaluate only:

- state status is `resting`
- minimum rest duration from the existing wake policy
- recovered fatigue at or below the existing wake threshold

Use the same trusted calculations as `run_wake_check`.

When blocked:

- do not invoke selector
- result is `remain_asleep`
- record deterministic blockers

### Selector decision model

Add a small normalized decision type with bounded text.

Allowed decisions:

```text
remain_asleep
select_venue
```

Allowed reason codes:

```text
no_specific_reason
rest_preferred
tie_unresolved
opaque_identity_changed
unresolved_impulse
comparison_unavailable
```

Accepted output:

```json
{
  "decision": "remain_asleep | select_venue",
  "selected_venue_id": "exact candidate ID or null",
  "observation": "bounded short text",
  "reason": "bounded short text",
  "reason_code": "bounded allowed code"
}
```

Validation:

- `remain_asleep` requires null selection
- `select_venue` requires one exact ID in the validated candidate set
- unsupported code or decision fails closed
- invalid or oversized fields fail closed
- no candidate ordering fallback
- deterministic mode always remains asleep

Rejected selector output must become a safe rejected `remain_asleep` result, not an unhandled crash.

### Command selector

Add:

```text
scripts/openai_compatible_wake_selector.py
```

Follow the existing wake-brain adapter's endpoint and credential safeguards, but use a selector-specific system prompt and schema.

The system prompt must state:

- this is not a Visit or wake command
- Venue content is unavailable and must not be inferred
- no tools, commands, URLs, writes, or automatic actions
- opaque identity difference is not knowledge of content change
- sleep is valid
- return exactly one JSON object

Selector payload may contain only:

```text
agent ID
bounded unresolved impulses
rest/fatigue facts and thresholds
sorted candidate metadata
output contract
```

It must not contain:

```text
memory text
profile prose
Venue labels
repository URLs
local paths
snapshot roots
source metadata
Venue or snapshot contents
brain command
report data
```

Use the same remote-endpoint API-key safeguard as the existing adapter.

### Persistent observation

Write one unique append-only record under:

```text
<agent>/wake_selections/YYYY-MM-DD_HHMMSS.json
```

Record at least:

- schema version
- agent ID
- checked time
- source: deterministic gate, deterministic default, or command selector
- eligibility and blockers
- policy and recovered rest facts
- registry path fingerprint or repository-relative identity without exposing it to published surfaces
- registry schema version
- sorted candidate facts
- safe current-snapshot omissions
- candidate validation status and safe error
- selector status, model label, protocol, and safe error
- final decision
- selected Venue ID or null
- bounded observation, reason, reason code
- `content_was_not_read: true`
- `wake_was_not_run: true`
- `request_was_not_created: true`
- `visit_was_not_run: true`
- local selection record path if consistent with existing local wake records

A selection record is the only persistent mutation allowed.

Do not write state, memory, impulse, Visit, wake, Trace, Request, report, or Gateway files.
